# 접근성 체크리스트

> 기준: **KWCAG 2.2** (KS X OT0003, 2022.12.28 개정) — 원칙 4개 / 지침 14개 / **검사항목 33개**
> 국내 법정 표준(『지능정보화기본법』, 『장애인차별금지법』)이므로 공공기관 프로젝트에서는 의무입니다.
> 참고 기준: WCAG 2.2 AA (글로벌 대응이 필요할 때)

**표기**
`[5.1.1]` KWCAG 2.2 검사항목 번호 — 법정 필수
`[권장]` KWCAG에는 없지만 실무상 지켜야 하는 것 (대부분 WCAG AA 수준)

**사용법**
컴포넌트를 만들 때는 해당 섹션만 보고, 프로젝트 종료 전에는 부록 A의 33개 인덱스로 전수 점검합니다.

## 1. 문서 기본 구조

템플릿(Starter)에 넣어두면 매 프로젝트에서 자동으로 통과하는 항목들입니다.

- [ ] `[7.1.1]` **기본 언어 명시**
  ```html
  <html lang="ko">
  ```
  일부 영역이 다른 언어면 해당 요소에도 지정: `<span lang="ja">東京</span>`

- [ ] `[6.4.2]` **페이지마다 고유한 `<title>`**
  - "OOO에 오신 것을 환영합니다"를 전 페이지에 쓰지 말 것
  - 팝업창·iframe에도 각각 제목 필요
  - 형식 예: `기관장 인사말 | OOO`

- [ ] `[6.4.1]` **건너뛰기 링크(skip navigation)**
  ```html
  <body>
    <a href="#main" class="skip-nav">본문 바로가기</a>
    ...
    <main id="main" tabindex="-1">
  ```
  ```css
  .skip-nav {
    position: absolute; left: 0; top: -100%;
    padding: 10px 20px; background: #000; color: #fff;
  }
  .skip-nav:focus { top: 0; }
  ```
  **초점을 받으면 화면에 보여야 합니다.** `sr-only`로 영원히 숨기면 위반입니다.

- [ ] `[6.4.2]` **제목(heading) 구조**
  - `h1` → `h2` → `h3` 순서, 레벨 건너뛰기 금지
  - 디자인상 글자가 작다는 이유로 `h2`를 `h4`로 쓰지 말 것 → CSS로 해결
  - 본문이 없는 블록에는 제목을 붙이지 않음

- [ ] `[권장]` **랜드마크 요소 사용**
  ```html
  <header> <nav> <main> <aside> <footer>
  ```
  `<div class="header">`가 아니라 실제 시맨틱 요소로. 스크린리더의 영역 간 이동에 쓰입니다.

- [ ] `[5.3.2]` **콘텐츠의 선형구조**
  - CSS를 끄고 봤을 때 읽는 순서가 논리적인지
  - `order`, `flex-direction: row-reverse`, `position: absolute`로 시각적 순서를 바꿨다면 DOM 순서와 어긋납니다
  - 확인: DevTools에서 CSS 전체 비활성화 후 위에서 아래로 읽어보기

- [ ] `[8.1.1]` **마크업 문법 오류 없음**
  - 태그 열고 닫음, 중첩 관계, 속성 중복 없음
  - **`id` 중복 금지** — React에서 컴포넌트를 반복 렌더할 때 자주 발생합니다
  - 확인: `validator.w3.org`

## 2. 이미지 · 미디어

- [ ] `[5.1.1]` **의미 있는 이미지에 적절한 대체 텍스트**
  ```html
  <img src="chart.png" alt="2026년 분기별 매출: 1분기 12억, 2분기 15억...">
  ```
  - `alt="이미지"`, `alt="img01"`은 없는 것과 같습니다
  - 이미지 링크·버튼은 **목적지나 기능**을 씀 (`alt="회사소개 페이지로 이동"`)
  - 복잡한 차트는 표나 본문으로 동등한 정보를 제공

- [ ] `[5.1.1]` **장식 이미지는 빈 대체 텍스트**
  ```html
  <img src="deco-line.png" alt="">
  <svg aria-hidden="true" focusable="false">...</svg>
  ```
  `alt` 속성 자체를 생략하면 안 됩니다. 스크린리더가 파일명을 읽습니다.

- [ ] `[5.1.1]` **의미 있는 배경 이미지에는 대체 수단**
  - CSS `background-image`로 넣은 텍스트 이미지는 스크린리더가 못 읽습니다
  - `sr-only` 텍스트를 함께 제공

- [ ] `[5.1.1]` **아이콘 버튼에 텍스트 대안**
  ```html
  <button type="button">
    <svg aria-hidden="true" focusable="false">...</svg>
    <span class="sr-only">메뉴 열기</span>
  </button>
  ```

- [ ] `[5.2.1]` **영상에 자막·대본·수어 중 하나 이상**
  - 음성만 있는 콘텐츠도 대본 필요
  - 대사 없이 영상만 있으면 화면해설 제공

- [ ] `[5.4.2]` **자동 재생 소리 금지**
  - 3초 이상 재생되는 소리는 정지·일시정지·음량조절 수단 필수
  - 배경음악 요구가 들어오면 이 항목을 근거로 제어 UI를 함께 요청하세요

- [ ] `[6.3.1]` **초당 3~50회 깜빡임·번쩍임 금지**
  - 화면 면적의 10%를 넘는 번쩍임 특히 주의
  - 광과민성 발작 유발 가능 — GSAP 플리커 효과 만들 때 확인

## 3. 색 · 시각적 표현

- [ ] `[5.4.3]` **텍스트 명도 대비 4.5:1 이상**
  - 18pt 이상 또는 14pt 굵은 글씨: 3:1까지 허용
  - 흰 배경에 `#999`는 미달(약 2.8:1), `#767676`부터 통과
  - 확인: DevTools 색상 피커, Colour Contrast Analyser

- [ ] `[권장]` **텍스트 아닌 요소도 3:1 이상**
  - 버튼 테두리, 입력창 경계, 체크박스, 아이콘, 그래프 등 (WCAG 1.4.11)
  - 연한 회색 테두리 입력창이 여기서 자주 걸립니다

- [ ] `[5.4.1]` **색만으로 정보 전달 금지**
  ```html
  <!-- ❌ 빨간 글씨로만 필수 표시 -->
  <label style="color:red">이름</label>
  <!-- ✅ -->
  <label>이름 <span class="required">(필수)</span></label>
  ```
  - 그래프는 색 + 무늬/패턴 또는 직접 라벨
  - 오류 표시는 색 + 아이콘 + 텍스트

- [ ] `[5.4.4]` **이웃한 콘텐츠 시각적 구분**
  - 테두리, 구분선, 배경색 차이, 충분한 간격 중 하나 이상
  - 카드가 배경과 같은 색이고 그림자도 없으면 경계를 인식할 수 없습니다

- [ ] `[5.3.3]` **지시사항이 감각에 의존하지 않을 것**
  - ❌ "오른쪽 버튼을 누르세요" / "빨간 버튼을 누르세요"
  - ✅ "저장 버튼을 누르세요"

- [ ] `[권장]` **200% 확대 시 콘텐츠 손실 없음**
  - 브라우저 줌 200%에서 텍스트 잘림·겹침·가로 스크롤 없는지 (WCAG 1.4.4)
  - 400% 확대 시 세로 스크롤만 발생 (WCAG 1.4.10)
  - 고정 높이 컨테이너에 `overflow: hidden`이 있으면 대부분 여기서 깨집니다

- [ ] `[권장]` **`user-scalable=no` 금지**
  ```html
  <!-- ❌ -->
  <meta name="viewport" content="width=device-width, user-scalable=no">
  <!-- ✅ -->
  <meta name="viewport" content="width=device-width, initial-scale=1">
  ```

## 4. 키보드 · 초점

**가장 많이 위반하고, 가장 쉽게 확인할 수 있는 영역입니다.**

- [ ] `[6.1.1]` **모든 기능을 키보드만으로 사용 가능**
  - 마우스를 치워두고 Tab / Shift+Tab / Enter / Space / 방향키 / ESC 만으로 처음부터 끝까지
  - 조작 횟수가 많은 건 문제되지 않음. **도달 불가능한 것**이 문제

- [ ] `[6.1.1]` **`div`/`span`에 클릭 이벤트 금지**
  ```html
  <!-- ❌ 키보드로 접근 불가 -->
  <div class="btn" onclick="...">확인</div>
  <!-- ✅ -->
  <button type="button">확인</button>
  ```
  부득이한 경우: `role="button"` + `tabindex="0"` + Enter/Space 키 핸들러를 **전부** 직접 구현해야 합니다. 그냥 `<button>`을 쓰세요.

- [ ] `[6.1.2]` **초점 표시가 항상 보일 것**
  ```css
  :root {
    --focus-ring: 2px solid var(--color-primary);
    --focus-offset: 2px;
  }
  .ui-button:focus-visible {
    outline: var(--focus-ring);
    outline-offset: var(--focus-offset);
  }
  ```
  - **`outline: none`만 쓰고 대체 스타일이 없으면 명백한 위반**입니다
  - `:focus-visible`을 쓰면 마우스 클릭 시에는 안 보이고 키보드 탐색 시에만 나타남 → 디자인 요구와 양립 가능

- [ ] `[6.1.2]` **초점 이동 순서가 논리적일 것**
  - 시각적 순서와 Tab 순서가 일치하는지
  - `tabindex`에 양수(1, 2, 3...) 사용 금지 — 전체 순서가 망가집니다. `0`과 `-1`만 사용

- [ ] `[6.1.2]` **키보드 트랩 없음**
  - 특정 요소에 들어간 뒤 Tab으로 빠져나올 수 없는 곳이 없는지
  - 에디터, 지도, 임베드 영역에서 자주 발생

- [ ] `[6.1.4]` **단일 문자 단축키 주의**
  - `S`, `/` 같은 단일 키 단축키를 제공한다면: 끌 수 있거나, 조합키로 재설정 가능하거나, 초점을 받았을 때만 동작해야 함

## 5. 폼 · 입력

- [ ] `[7.3.2]` **모든 입력에 프로그래밍적으로 연결된 레이블**
  ```html
  <label for="email">이메일</label>
  <input id="email" type="email">
  ```
  - **placeholder는 레이블이 아닙니다.** 입력 시작하면 사라지고, 대비도 낮습니다
  - 시각적으로 레이블을 숨겨야 한다면 `sr-only` 사용 (`display:none`은 스크린리더도 못 읽음)

- [ ] `[7.3.1]` **오류의 위치와 이유를 텍스트로 안내**
  ```html
  <div class="ui-field">
    <label for="email">이메일</label>
    <input id="email" type="email"
           aria-describedby="email-help email-error"
           aria-invalid="true">
    <p id="email-help" class="ui-help">회사 메일을 입력하세요</p>
    <p id="email-error" class="ui-error" role="alert">
      올바른 이메일 형식이 아닙니다
    </p>
  </div>
  ```
  - "입력값을 확인하세요"만으로는 부족 — **어떤 항목이 왜** 틀렸는지
  - `role="alert"`로 스크린리더에 즉시 전달

- [ ] `[6.5.3]` **레이블과 네임 일치**
  ```html
  <!-- ❌ 화면엔 "검색", 접근성 이름은 "찾기" → 음성명령 사용자가 실행 불가 -->
  <button aria-label="찾기">검색</button>
  <!-- ✅ -->
  <button>검색</button>
  ```
  `aria-label`이 화면 텍스트를 덮어쓰지 않도록 주의. 화면 텍스트가 있으면 `aria-label`을 붙이지 않는 게 기본입니다.

- [ ] `[7.2.1]` **초점·선택만으로 기능이 실행되지 않을 것**
  - 셀렉트에서 항목을 고르는 것만으로 페이지 이동 금지 → 별도 "확인" 버튼
  - 마지막 칸 입력 시 자동 제출 금지
  - 새 창은 사용자가 예측 가능한 상황에서만 (링크 텍스트에 "새 창" 명시)

- [ ] `[7.3.3]` **인지 기능 테스트에만 의존하지 않는 인증 수단**
  - 비밀번호 입력, 패턴 기억, 이미지 찾기 CAPTCHA 등 외에 대안 하나 이상
  - 브라우저 자동완성 허용(`autocomplete` 속성), OAuth, 생체·기기 인증 등

- [ ] `[7.3.4]` **반복 입력 정보 자동/선택 입력**
  - "주문자 정보와 동일" 체크박스 등
  - 보안상 재입력이 필요한 비밀번호는 예외

- [ ] `[권장]` **`autocomplete` 속성 지정**
  ```html
  <input type="email" autocomplete="email">
  <input type="tel" autocomplete="tel">
  ```

## 6. 컴포넌트별

### 모달 / 레이어 팝업

- [ ] 열 때 초점을 모달 내부로 이동, **닫을 때 원래 트리거로 복귀** `[6.1.2]`
- [ ] 포커스 트랩 — Tab이 모달 밖으로 나가지 않음 `[6.1.2]`
- [ ] ESC로 닫힘 `[7.2.1]`
- [ ] 배경 스크롤 락
- [ ] 마크업
  ```html
  <div role="dialog" aria-modal="true" aria-labelledby="modal-title">
    <h2 id="modal-title">알림</h2>
    ...
    <button type="button" class="modal-close">
      <span class="sr-only">닫기</span>
    </button>
  </div>
  ```
- [ ] 배경 콘텐츠에 `inert` 또는 `aria-hidden="true"` `[권장]`

> KWCAG 7.2.1은 레이어 팝업 사용 자체를 권장하지 않습니다. 쓸 거면 위 조건을 전부 지켜야 합니다.

### 탭

- [ ] 방향키로 탭 간 이동, Tab 키로는 탭 목록 전체를 하나로 취급
- [ ] `role="tablist"` / `role="tab"` / `role="tabpanel"`, `aria-selected`, `aria-controls`

### 아코디언

- [ ] `<button aria-expanded="true|false" aria-controls="panel-1">`
- [ ] 닫힌 패널은 `hidden` 또는 `display:none` (초점이 숨은 요소로 들어가지 않도록)

### 드롭다운 메뉴 / GNB

- [ ] hover뿐 아니라 **키보드 초점으로도 하위 메뉴 열림** `[6.1.1]`
- [ ] 마우스가 벗어나도 초점이 있으면 유지 `[권장]`
- [ ] ESC로 닫히고 초점은 상위 버튼으로

### 토스트 / 상태 메시지

- [ ] `role="status"` (일반) 또는 `role="alert"` (긴급) `[권장]`
- [ ] 자동으로 사라진다면 충분한 시간 또는 정지 수단 `[6.2.1]`

### 캐러셀 / 자동 슬라이드

- [ ] **정지 버튼 제공** `[6.2.2]` — 자동 롤링 배너는 이 항목 없이는 통과 불가
- [ ] 이전/다음 버튼에 텍스트 대안
- [ ] 화면에 안 보이는 슬라이드는 초점을 받지 않도록

### 표(테이블)

- [ ] `[5.3.1]` 제목 셀과 데이터 셀 구분
  ```html
  <table>
    <caption>2026년 분기별 매출</caption>
    <thead>
      <tr><th scope="col">분기</th><th scope="col">매출</th></tr>
    </thead>
    <tbody>
      <tr><th scope="row">1분기</th><td>12억</td></tr>
    </tbody>
  </table>
  ```
- [ ] 레이아웃 목적의 `<table>` 사용 금지

## 7. 모바일 · 터치 · 동작

- [ ] `[6.1.3]` **컨트롤 크기 확보**
  - 대각선 6.0mm 이상 (실무상 44×44 CSS px 이상 권장)
  - 이웃한 컨트롤이 같이 눌리지 않을 만큼의 간격

- [ ] `[6.5.1]` **다중 포인터·경로기반 동작의 단일 포인터 대안**
  - 핀치 줌, 두 손가락 스와이프, 드래그앤드롭으로만 되는 기능이 있다면 버튼 등 대안 제공
  - 스와이프 슬라이더에는 이전/다음 버튼 필수

- [ ] `[6.5.2]` **포인터 입력 취소 가능**
  - `mousedown`/`touchstart`만으로 기능이 실행되지 않도록 (`click` 사용)

## 부록 C. 자주 틀리는 것 Top 10

진단해보면 대체로 이 목록 안에서 나옵니다. 새 프로젝트 시작 시 이 10개만 먼저 확인해도 절반은 예방됩니다.

1. `outline: none`만 있고 대체 초점 스타일 없음 — `[6.1.2]`
2. `alt="이미지"`, `alt="img01"` 같은 무의미한 대체 텍스트 — `[5.1.1]`
3. placeholder를 레이블 대신 사용 — `[7.3.2]`
4. `div`/`span`에 클릭 이벤트 — `[6.1.1]`
5. 흰 배경에 `#999` 회색 텍스트 (명도 대비 미달) — `[5.4.3]`
6. 디자인 때문에 제목 레벨 건너뜀 (`h2` → `h4`) — `[6.4.2]`
7. 자동 롤링 배너에 정지 버튼 없음 — `[6.2.2]`
8. 모달 열어도 초점이 뒤에 남아있음 — `[6.1.2]`
9. **"자세히 보기", "더보기", "여기를 클릭"이 페이지에 20개** — `[6.4.3]`
   → 링크 텍스트만으로 목적지를 알 수 있게 하거나, 최소한 맥락(주변 제목)에서 유추 가능해야 합니다. `sr-only`로 보완:
   ```html
   <a href="/notice">더보기<span class="sr-only"> (공지사항)</span></a>
   ```
10. 전 페이지 `<title>`이 동일 — `[6.4.2]`
